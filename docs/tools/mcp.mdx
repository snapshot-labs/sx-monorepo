The **Snapshot MCP server** lets any [MCP](https://modelcontextprotocol.io)-compatible AI assistant read governance data and **act on your behalf**: cast votes, create proposals, follow spaces. All writes are signed by a per-user alias wallet that you authorize once and can revoke at any time.

The hosted server is available at **`https://mcp.snapshot.box`**. No install or API key required for read access.

```text
You: "Vote 'For' on the active ENS proposal about treasury rebalancing."
AI:  → snapshot-query    (find the proposal)
     → snapshot-vote     (sign and submit via your authorized alias)
     ✓ Voted.
```

| Client | What to do |
|---|---|
| **Claude.ai** | Settings > Integrations > Add integration > `https://mcp.snapshot.box` |
| **Claude Desktop** | Edit `claude_desktop_config.json` ([details](#claude-desktop)) > restart |
| **Claude Code** | `claude mcp add snapshot --transport http https://mcp.snapshot.box` |
| **ChatGPT** | Settings > Connectors > Connect to MCP > `https://mcp.snapshot.box` |
| **Codex CLI** | Add `[mcp_servers.snapshot]` to `~/.codex/config.toml` ([details](#codex)) |
| **Cursor** | Settings > MCP > Add > URL `https://mcp.snapshot.box` |

The first time the AI calls a write tool, an OAuth window opens so you can authorize a Snapshot alias against your main wallet. This is a one-time step, and you can revoke the alias at any time.

Once connected, try: *"What proposals are active in `ens.eth` right now?"*

## Authorize and revoke an alias

When you first use a write tool (vote, propose, follow), the MCP server redirects you to Snapshot to authorize an alias wallet. The alias is a separate wallet managed by the MCP server that can only sign Snapshot governance messages on your behalf. Your main wallet key never leaves your wallet.

Once authorized, the alias can:
- Cast a vote
- Publish a proposal
- Follow a space

You can revoke an alias at any time. Either open the authorize link (`snapshot.box/#/settings/alias/authorize/<alias-address>`) while connected with your main wallet and click **Revoke alias**, or go to **Settings > Aliases** to see and revoke all your active aliases.

## Try it

A few prompts to validate the connection:

- *"What proposals are active in `ens.eth` right now?"*
- *"What's the current voting power of `vitalik.eth` in `ens.eth`?"*
- *"Summarize the last 5 closed proposals on `aave.eth` and tell me which passed."*
- *"Follow `gitcoindao.eth` for me."*
- *"Vote 'For' on proposal `0xabc...` in `ens.eth`."*
- *"Create a proposal in `myspace.eth` titled 'Q3 budget' with body 'Allocate $50k...'."*

The AI chains tools automatically: `snapshot-query` to discover data, then `snapshot-vote`, `snapshot-propose`, or `snapshot-follow` to act.

## Tools

The server exposes 5 tools. The AI selects them based on your intent.

| Tool | Purpose | Auth required |
|---|---|---|
| `snapshot-query` | Run any GraphQL query against Snapshot. Auto-binds your address as `$user`. | No |
| `snapshot-schema` | Return the GraphQL schema. The AI uses this when a query fails on an unknown field. | No |
| `snapshot-vote` | Cast a vote. Voting type and privacy are auto-detected from the proposal. Calling again on the same proposal replaces the previous vote. | Yes |
| `snapshot-propose` | Create a proposal. Only `space`, `title`, and `body` are required. Voting type, choices, period, and privacy default from the space settings. | Yes |
| `snapshot-follow` | Follow a space. Calling again on a space you already follow is a no-op. | Yes |

<Tip>
The full input schema for each tool is exposed via standard MCP discovery, so your AI client introspects them automatically.
</Tip>

## How it works

1. **Discover.** The client fetches `/.well-known/oauth-authorization-server` and the MCP manifest to learn endpoints and tools.
2. **Connect (one-time).** OAuth 2.0 with PKCE. The server creates a fresh CDP-managed alias wallet, redirects you to Snapshot to authorize that alias against your main wallet, then issues a JWT tied to your address and alias.
3. **Read.** `snapshot-query` and `snapshot-schema` proxy to Snapshot Hub. No wallet needed.
4. **Write.** Write tools build the EIP-712 envelope, ask CDP to sign with your alias, and submit to the Snapshot sequencer.

The server is **stateless**. It stores no per-user data. The JWT carries everything (address, CDP account ID, alias).

## Security

| Property | How it's enforced |
|---|---|
| **Your main key never leaves your wallet** | Writes are signed by an *alias*, a separate wallet you authorize once via Snapshot's alias flow. The alias can only sign Snapshot governance messages. |
| **One alias per user, isolated** | Each OAuth connection provisions its own [Coinbase CDP](https://docs.cdp.coinbase.com)-managed alias. The server never sees the private key. CDP signs on demand using the per-user account ID embedded in the JWT. |
| **Tokens are scoped** | Access tokens are stateless JWTs (HS256). A stolen token can act *as that alias* but cannot touch your main wallet or impersonate other users. Rotating `JWT_SECRET` invalidates all tokens. |
| **OAuth with PKCE** | Standards-compliant Authorization Code + PKCE flow. No client secret required for native and desktop clients. |
| **Anonymous reads** | The OAuth flow only triggers when you use a write tool. Read-only queries never need auth. |
| **No data stored** | The server keeps zero session state. Every request is processed from the JWT alone. |
| **Revocable** | You can [revoke the alias](#revoke-an-alias) at any time from the Snapshot UI. |

## Auth flow

<Tabs>
<Tab title="HTTP (OAuth 2.0 + PKCE)">

1. Client calls `/authorize`.
2. Server creates a fresh CDP alias wallet.
3. Server redirects to `snapshot.box/#/settings/alias/authorize/<alias>` with a callback URL.
4. You sign the alias authorization with your main wallet.
5. Snapshot redirects back to `/auth/callback`.
6. Server resolves the authorizing user from the alias mapping, mints a JWT, and redirects back to the client.
7. Client exchanges the authorization code for a JWT at `/token`.
8. All subsequent requests carry the JWT. The server reads `{ address, aliasAccountId, alias }` from claims and asks CDP to sign.

</Tab>
<Tab title="Stdio (local key)">

1. Call any write tool. If the alias key isn't yet authorized, the tool returns a URL to `snapshot.box/#/settings/alias/authorize/<alias>`.
2. Authorize the alias once in your browser.
3. Subsequent calls succeed.

</Tab>
</Tabs>

## Self-hosting

Run your own instance from the monorepo:

```bash
git clone https://github.com/snapshot-labs/sx-monorepo
cd sx-monorepo
bun install

cp apps/mcp/.env.example apps/mcp/.env   # fill in JWT + CDP creds
cd apps/mcp && bun start                 # HTTP on :8080
```

Requirements: [Bun](https://bun.sh) 1.0 or later, a [Coinbase CDP](https://portal.cdp.coinbase.com) project (free tier works), and a 32-byte `JWT_SECRET` (`openssl rand -hex 32`).

### Configuration

| Variable | Required for | Description |
|---|---|---|
| `JWT_SECRET` | HTTP | HS256 secret, 32+ chars. Rotate to invalidate all tokens. |
| `CDP_API_KEY_ID` / `CDP_API_KEY_SECRET` / `CDP_WALLET_SECRET` | HTTP | Coinbase CDP credentials for per-user alias wallets. |
| `BASE_URL` | HTTP | Public URL for OAuth metadata (e.g. `https://mcp.snapshot.box`). |
| `PORT` | any | HTTP port (default `8080`). |
| `ALIAS_PRIVATE_KEY` | stdio | A single Snapshot alias key. Ignored by HTTP. |
| `SNAPSHOT_API_KEY` | any | [API key](/tools/api/api-keys) for higher rate limits. |
| `SNAPSHOT_API_URL` | any | GraphQL endpoint (defaults to mainnet hub). |

### Stdio for local development

```bash
cd apps/mcp
ALIAS_PRIVATE_KEY=0x... bun src/index.ts --stdio
```

Useful for scripting, CI, or testing changes locally before deploying.

## FAQ

<AccordionGroup>
<Accordion title="Do I need to give the AI my private key?">
No. You authorize a Snapshot *alias*, a separate signer managed by the MCP server. Your main wallet key stays in your wallet.
</Accordion>

<Accordion title="Can the AI drain my funds?">
No. The alias can only sign Snapshot governance messages (votes, proposals, follows). It cannot move tokens, call other contracts, or change your wallet settings.
</Accordion>

<Accordion title="How do I revoke access?">
Open the alias authorization page on Snapshot (`snapshot.box/#/settings/alias/authorize/<alias-address>`) while connected with your main wallet, and click **Revoke alias**. This removes the alias authorization. You can also disconnect the MCP connector in your AI client to stop it from making requests, though that does not revoke the alias itself.
</Accordion>

<Accordion title="Is the read API rate-limited?">
Yes, the standard Snapshot rate limit applies. Self-host with your own `SNAPSHOT_API_KEY` for higher throughput.
</Accordion>

<Accordion title="What does the server store?">
Nothing. The server is stateless. Tokens are self-verifiable JWTs.
</Accordion>
</AccordionGroup>
